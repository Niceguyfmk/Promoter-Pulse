Promoter Pulse Architecture

A comprehensive architectural analysis and database specification for enterprise field force management.

1. Executive Summary

Promoter Pulse is a scalable multi-tenant platform designed to solve the complexities of retail field marketing. By integrating real-time geolocation tracking with structured data collection, it provides organizations with a "single source of truth" for field operations.

Key Pillars

  • Granular Multi-Tenancy: Native Postgres RLS.
  • Role-Based Access: Promoter, Manager, Admin layers.
  • Field Proofing: GPS-verified check-ins.

Business Impact

  • Reduction in attendance fraud.
  • Real-time merchandising visibility.
  • Automated reporting pipelines.

2. Technical Stack & Patterns

The Core Stack

  • Framework: Next.js 14+ (App Router)
  • Backend-as-a-Service: Supabase
  • Database: PostgreSQL 15
  • Auth: Supabase GoTrue (JWT/RLS)

Architectural Patterns

  • Feature-Sliced Design: Modular functionality.
  • Server Actions: Secure database mutation.
  • Shared Client Context: Real-time UI updates.

5. Information Systems Analysis

System Modules

  1. Identity (IAM): Handles invite-only onboarding. Admins generate metadata-linked invites that the system consumes to auto-provision users.
  2. Operations (Shift Management): The logic engine for scheduling. It calculates compliance based on scheduled_start_at vs checked_in_at.
  3. Field Intel: The data collection layer. Uses Postgres JSONB to allow flexible form schemas without migrations.

7. Multi-Tenancy & Security

Row Level Security (RLS)

Isolation is not handled in application code, but at the Database Engine level. This prevents "Leaky Abstractions" where one tenant could accidentally see another's data.

CREATE POLICY tenant_isolation ON shifts 
FOR ALL USING (tenant_id = current_app_tenant_id());

Auth Model: Integrated with Supabase Auth. A Postgres trigger handle_new_user() manages the mapping between the Auth identity and the Business Identity.

8. Architectural Risks & Recommendations

Risks

  • Status Consistency: visit_reports uses text checks instead of enums.
  • Storage Coupling: Photo metadata is tightly coupled to shifts, making ad-hoc photo audits harder to query.

Recommendations

  • Reporting Engine: Implement materialized views for daily attendance aggregation.
  • Indexes: Add partial indexes for deleted_at to speed up filtering.
